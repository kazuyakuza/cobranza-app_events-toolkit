# Propose `ManifestContributor` pattern for `@cobranza-apps/events-toolkit`

## Initial Note

This file is related to the problem described below, so [task](#task) section is a hole task.

## Problem

Currently, microservices that register subjects dynamically at runtime (like `ms-db-gateway` with 154 generic CRUD subjects + 1 transaction subject = 155 dynamic entries) must manually patch `DiscoveryService.getManifest().consumes`. This is fragile and error-prone.

In `ms-db-gateway` this currently takes the form of `DiscoveryManifestPatchService` (manual patch) coupled with `CrudSubjectManifestContributor` (155 dynamic subjects). The manual patch introduces a lifecycle ordering bug: the library `DiscoveryService` publishes `platform.service.register.v1` from `onApplicationBootstrap` **before** the patch service mutates the cached manifest, so the canonical registration event omits the dynamic entries. The root cause is the absence of a library-sanctioned contributor extension point.

## Proposed Solution

Add a `ManifestContributor` interface to `@cobranza-apps/events-toolkit`:

```typescript
interface ManifestContributor {
  contributeConsumes(): ManifestConsumeEntry[];
  contributeProduces(): ManifestProduceEntry[];
}
```

## Usage

- Services implement `ManifestContributor` and register themselves with `DiscoveryService`.
- `DiscoveryService` calls all registered contributors during `onModuleInit` before building the manifest.
- This ensures dynamic entries are included in schema generation and in the `platform.service.register.v1` event.

## Benefits

1. No manual manifest patching.
2. Dynamic entries participate in schema generation.
3. Lifecycle ordering handled by the library.
4. Cleaner API for services with runtime-generated handlers.

## Implementation notes

- Add `registerContributor(contributor: ManifestContributor)` method to `DiscoveryService`.
- Call `contributors.flatMap((c) => c.contributeConsumes())` when building the manifest.
- Ensure contributors are called **before** `generateSchemasForManifest()` and **before** the registration event is published.
- Document the new extension point in `event-discovery-and-service-registry.md`.

## Context for ms-db-gateway

`ms-db-gateway` currently uses `DiscoveryManifestPatchService` (manual patch) with `CrudSubjectManifestContributor` (155 dynamic subjects). Migrating to the new `ManifestContributor` pattern would eliminate the lifecycle bug where registration is published before the patch runs, and would remove the need for `DiscoveryManifestPatchService` entirely.
