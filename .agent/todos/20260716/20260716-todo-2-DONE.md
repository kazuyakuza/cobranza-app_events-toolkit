# Modify @cobranza-apps/events-toolkit EventEnvelope Validation

## Context

The `docs/manual-nats-testing.md` guide (from `ms-db-gateway`) was found to document an incomplete `EventEnvelope` structure. Investigation revealed that `@cobranza-apps/events-toolkit@0.11.6` enforces a strict wire contract with 10 required fields (`id`, `type`, `version`, `produced_at`, `producer`, `company_id`, `actor_type`, `actor_id`, `correlation_id`, `data`).

The user wants to relax two specific validation rules in the toolkit library and introduce a new envelope variant:

## Proposed Changes to @cobranza-apps/events-toolkit

### Change 1: Make `actor_id` optional for non-human actor types

**Current behavior:** `actor_id` is required for ALL actor types (`client`, `company_user`, `system`, `scheduler`, `external_api`). It must be a non-empty string (`@IsString()` + `@IsNotEmpty()`).

**Desired behavior:** `actor_id` should be optional when `actor_type` is `system`, `scheduler`, or `external_api`. These actor types represent automated/non-human processes that don't correspond to real database users.

**Implementation approach:**

- Add `@ValidateIf((o) => ['client', 'company_user'].includes(o.actor_type))` decorator to `actor_id` in `EventEnvelope`
- OR: better add `@IsOptional()` with a custom validator that checks `actor_type`, like `@IsOptionalForSystemActors` (export it so anyone that imports the lib can use it).

### Change 2: Introduce a "Global Event" envelope type for operations without a tenant

**Current behavior:** `company_id` is required for ALL events. It must be a UUID v4 (`@IsUUID('4')`). The toolkit's NATS subject builder embeds `companyId` into routing keys (`company.${companyId}....`).

**Desired behavior:** Introduce a separate `GlobalEventEnvelope` class (or `SystemEventEnvelope`) that omits `company_id` entirely. This is for operations that are not tenant-scoped, such as:

- Creating global entities (`company`, `user`, `role`)
- System-wide configuration changes
- Cross-tenant aggregate queries (when the caller has system-wide privileges)

**Rationale:**

- `company` is a global entity — it cannot have a `companyId` field because it IS the company
- The current workaround (sending a placeholder UUID as `company_id`) is semantically incorrect and confusing
- A separate envelope type makes the intent explicit and prevents accidental tenant-scoped routing for global operations

**Implementation approach:**

- Create a new `GlobalEventEnvelope` class that extends a shared base (or is a separate type) without `company_id`
- Update the toolkit's consumer/producer to accept either `EventEnvelope` or `GlobalEventEnvelope`
- Update NATS subject routing for global events (e.g., `system.db.company.create.v1` or `global.db.company.create.v1`)
- Consider how `EventContext` should handle the absence of `companyId`
