# Fix: @cobranza-apps/events-toolkit EventsToolkitModule.forRootAsync missing exports

## Problem

`EventsToolkitModule.forRootAsync` creates internal providers but does not export them. Its imported sub-modules (`ProducerModule`, `ConsumerModule`, `OutboxModule`, `DiscoveryModule`) cannot resolve these dependencies, causing NestJS DI errors in any consumer that uses the async registration path.

**Confirmed affected consumer:** `ms-db-gateway` (via `NatsModule` → `EventsToolkitModule.forRootAsync`).

## Root Cause

In `dist/events-toolkit.module.js`, `forRootAsync` returns:

```javascript
return {
    module: EventsToolkitModule_1,
    global: true,
    imports,
    providers: [optionsProvider, jetStreamProvider, loggingProvider],
};
```

The dynamic module has **no `exports` array**. In NestJS, a child module can only resolve providers from its parent module if they are explicitly exported. Since `ProducerModule` (imported by `EventsToolkitModule`) needs `JETSTREAM_TOKEN` (`"NATS_JETSTREAM"`) and `EventLoggerService`, and `ConsumerModule`/`OutboxModule`/`DiscoveryModule` need `EVENTS_TOOLKIT_OPTIONS`, they all fail with dependency resolution errors.

The synchronous `forRoot` path is unaffected because it passes resolved values directly into the sub-modules (e.g., `ProducerModule.forRoot({ jetStream })` creates its own `JETSTREAM_TOKEN` provider). Only the async `forRootAsync` path is broken.

## Fix

Add `exports` array to the dynamic module returned by `EventsToolkitModule.forRootAsync`:

```javascript
return {
    module: EventsToolkitModule_1,
    global: true,
    imports,
    providers: [optionsProvider, jetStreamProvider, loggingProvider],
    exports: [optionsProvider, jetStreamProvider, loggingProvider],
};
```
