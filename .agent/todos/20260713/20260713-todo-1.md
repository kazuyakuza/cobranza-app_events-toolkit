# Fix: @cobranza-apps/events-toolkit — Comprehensive DI Resolution Audit

## Background

The `@cobranza-apps/events-toolkit` library has multiple DI resolution bugs in its `forRootAsync` path. These bugs cause NestJS dependency injection failures in any consumer that uses the async registration path (`EventsToolkitModule.forRootAsync`).

**Confirmed affected consumer:** `ms-db-gateway`.

## Bugs Found

### Bug 1: `EventsToolkitModule.forRootAsync` missing exports (FIXED in 0.10.2)

`forRootAsync` creates providers `EVENTS_TOOLKIT_OPTIONS`, `JETSTREAM_TOKEN`, and `EventLoggerService` but does not export them. Sub-modules (`ProducerModule`, `ConsumerModule`, `OutboxModule`, `DiscoveryModule`) cannot resolve these dependencies.

**Fix:** Add `exports: [optionsProvider, jetStreamProvider, loggingProvider]` to the dynamic module returned by `forRootAsync`.

### Bug 2: `DiscoveryModule` missing `@nestjs/core` DiscoveryModule import (STILL BROKEN in 0.10.2)

`DiscoveryModule.forRoot` and `DiscoveryModule.forRootAsync` provide `ManifestServiceDepsProvider` which injects:

- `MetadataScanner` (constructor param)
- `DiscoveryService` (property decorator `@Inject(DiscoveryService)`)
- `Reflector` (property decorator `@Inject(Reflector)`)

All three are provided by `@nestjs/core`'s `DiscoveryModule`, but the library's `DiscoveryModule` never imports it.

**Error:**

```
Nest can't resolve dependencies of the ManifestServiceDepsProvider (?).
Please make sure that the argument MetadataScanner at index [0] is available in the DiscoveryModule module.
```

**Fix (both `forRoot` and `forRootAsync`):**

```typescript
import { DiscoveryModule as NestDiscoveryModule } from '@nestjs/core';

// In forRoot return:
return {
    module: DiscoveryModule,
    global: true,
    providers,
    exports: exported,
    imports: [NestDiscoveryModule],
    controllers: [DiscoveryController],
};

// In forRootAsync return:
return {
    module: DiscoveryModule,
    global: true,
    providers,
    exports: exported,
    imports: [NestDiscoveryModule, ...(asyncOptions.imports ?? [])],
    controllers: [DiscoveryController],
};
```

## Root Cause

The library was only tested with the synchronous `forRoot` path, where resolved values are passed directly to sub-modules. The async `forRootAsync` path has never been fully end-to-end tested.

## Fix Strategy

1. Update `DiscoveryModule.forRoot` and `forRootAsync` to import `@nestjs/core`'s `DiscoveryModule`
2. Add an end-to-end integration test that compiles `EventsToolkitModule.forRootAsync` with all sub-modules enabled (producer, consumer, outbox, discovery)
3. Ensure the test mocks `NATS_CONNECTION` but exercises the full DI graph
