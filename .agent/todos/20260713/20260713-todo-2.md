# Fix: @cobranza-apps/events-toolkit — Comprehensive DI Resolution Audit (All Bugs)

## Background

The `@cobranza-apps/events-toolkit` library has multiple DI resolution bugs in its `forRootAsync` path. These bugs cause NestJS dependency injection failures in any consumer that uses the async registration path (`EventsToolkitModule.forRootAsync`).

**Confirmed affected consumer:** `ms-db-gateway`.

**Root cause:** The library was only tested with the synchronous `forRoot` path, where resolved values are passed directly to sub-modules. The async `forRootAsync` path has never been fully end-to-end tested.

## Bugs Found

### Bug 1: `EventsToolkitModule.forRootAsync` missing exports (FIXED in 0.10.2)

`forRootAsync` creates providers `EVENTS_TOOLKIT_OPTIONS`, `JETSTREAM_TOKEN`, and `EventLoggerService` but does not export them. Sub-modules (`ProducerModule`, `ConsumerModule`, `OutboxModule`, `DiscoveryModule`) cannot resolve these dependencies.

**Fix:** Add `exports: [optionsProvider, jetStreamProvider, loggingProvider]` to the dynamic module returned by `forRootAsync`.

### Bug 2: `DiscoveryModule` missing `@nestjs/core` DiscoveryModule import (FIXED in 0.10.3)

`DiscoveryModule.forRoot` and `DiscoveryModule.forRootAsync` provide `ManifestServiceDepsProvider` which injects `MetadataScanner`, `DiscoveryService`, and `Reflector` from `@nestjs/core`, but the library's `DiscoveryModule` never imports it.

**Fix:** Add `imports: [NestDiscoveryModule]` (from `@nestjs/core`) to both `DiscoveryModule.forRoot` and `forRootAsync`.

### Bug 3: `RequestReplyService` never registered as provider (STILL BROKEN in 0.10.3)

`RequestReplyService` is exported from `dist/request-reply/index.js` and has `@Injectable()`, but it is **never added to any module's `providers` or `exports` array**. Consumers like `CrudHandlersModule` that inject `RequestReplyService` fail with DI errors.

**Error:**

```
Nest can't resolve dependencies of the Symbol(GenericCrudHandlerDeps) (RequestReplyConsumerService, ?).
Please make sure that the argument RequestReplyService at index [1] is available in the CrudHandlersModule module.
```

**Fix:** Register `RequestReplyService` and its dependency token `REQUEST_REPLY_DEPS_TOKEN` in `EventsToolkitModule.forRootAsync` providers and exports. Or create a `RequestReplyModule` that owns and exports `RequestReplyService`.

`RequestReplyService` dependencies (via `REQUEST_REPLY_DEPS_TOKEN`):

- `natsConnection` — NATS connection
- `producerService` — `ProducerService` (from `ProducerModule`)
- `logger` — `EventLoggerService`
- `config` — request-reply config (optional, can default)

## Fix Strategy

1. Update `EventsToolkitModule.forRootAsync` to provide and export `RequestReplyService` and `REQUEST_REPLY_DEPS_TOKEN`
2. Verify all 3 bugs are fixed end-to-end
3. Add a comprehensive end-to-end integration test that compiles `EventsToolkitModule.forRootAsync` with all sub-modules enabled (producer, consumer, outbox, discovery, request-reply)
4. Ensure the test mocks `NATS_CONNECTION` but exercises the full DI graph
