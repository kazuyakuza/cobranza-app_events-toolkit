# Fix `@cobranza-apps/events-toolkit` â€” `@jest/globals` Leaked Into Main Entry

IMPORTANT: this is an issue found in a project that imports `@cobranza-apps/events-toolkit` library. Take this TODO file as a hole task.

## Problem [DONE]

The `@cobranza-apps/events-toolkit` package main entry point (`dist/index.js`) transitively imports `dist/testing/assertion.helpers.js`, which requires `@jest/globals`. This causes any consumer that imports from the main entry outside a Jest test environment to crash with:

```
Do not import `@jest/globals` outside of the Jest test environment
```

## Impact

- **Consumers cannot safely import `ActorType`, `RequestContext`, or any main-entry export** in contexts like:
  - Jest `globalSetup` / `globalTeardown`
  - NestJS CLI scripts, seeders, or migration runners
  - Any Node process that doesn't run inside Jest's test environment
- In `ms-db-gateway`, this blocked all test execution because `globalSetup` loaded TypeORM entities, which imported `ActorType` from the toolkit.

## Root Cause

- `dist/events-toolkit.module.js` (or `dist/index.js`) has a dependency chain that pulls in `dist/testing/index.js`.
- `dist/testing/assertion.helpers.js` contains `require('@jest/globals')` (line 7).
- Testing utilities should be isolated behind a dedicated subpath export (e.g., `@cobranza-apps/events-toolkit/testing`), not reachable from the main module graph.

## Suggested Fix

1. **Audit the main module graph:**
   - Trace why `dist/events-toolkit.module.js` â†’ `dist/testing/index.js` exists.
   - Likely causes:
     - `EventsToolkitModule` imports something from a file that also exports testing utilities.
     - A barrel (`index.ts`) combines runtime and testing exports.

2. **Isolate testing exports:**
   - Remove all testing-only imports from the main module graph.
   - Expose testing helpers exclusively through a subpath export:

     ```json
     "exports": {
       ".": "./dist/index.js",
       "./testing": "./dist/testing/index.js"
     }
     ```

   - Update internal imports so `dist/testing/` is only referenced via the `./testing` subpath.

3. **Verify the fix:**
