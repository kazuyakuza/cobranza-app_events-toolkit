## When to Use the Outbox Pattern

The Outbox Pattern is not determined by whether communication is synchronous (Request/Reply) or asynchronous (Async Request/Response). The deciding factor is whether you need to guarantee consistency between a database change and a message publication.

### Request/Reply (Synchronous)

Example:

```text
Service A -> request -> Service B
Service A <- response <- Service B
```

If Service A simply sends a request and waits for a response, without persisting any related state locally, the Outbox Pattern is typically unnecessary.

Example:

```ts
const user = await nc.request("user.get", payload);
```

If the request fails or times out, the operation fails and can be retried or reported as an error. This is conceptually similar to an HTTP or gRPC call.

### Database Change + Message Publication

Example:

```ts
await db.insert(order);
await nc.publish("order.created");
```

This introduces a consistency problem:

```text
DB insert succeeds
Message publish fails
```

The order exists in the database, but no downstream service is notified.

In this scenario, the Outbox Pattern should be used:

```text
BEGIN TRANSACTION

INSERT order
INSERT outbox_event

COMMIT
```

A separate publisher process later reads the outbox table and publishes the event reliably.

### Async Request/Response

Example:

```ts
await db.insert(job);
await nc.publish("resume.parse.request");
```

This has the exact same consistency problem:

```text
Job inserted successfully
Message publish fails
```

The job remains in a pending state forever because no worker receives the request.

The recommended approach is:

```text
BEGIN TRANSACTION

INSERT job
INSERT outbox_event

COMMIT
```

Then a background publisher sends the message from the outbox.

## Practical Rule

Use the Outbox Pattern whenever an operation requires:

```text
Database state change
+
Message publication
```

and both must succeed consistently.

Typical cases where Outbox is recommended:

- Domain events (`order.created`, `payment.completed`)
- Integration events
- Async commands (`resume.parse.request`, `image.generate.request`)
- Any workflow that persists state and then publishes a message

Typical cases where Outbox is usually unnecessary:

- Pure Request/Reply interactions
- Read-only queries
- Synchronous service calls that do not modify local state

## Key Insight

The Outbox Pattern does not protect the message broker.

It protects consistency between:

```text
Your Database
and
Your Message Broker
```

If no database change is involved, there is usually no consistency problem for the Outbox Pattern to solve.
