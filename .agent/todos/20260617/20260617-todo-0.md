# TODO List #6 – Event Discovery Module

Library: `cobranza-apps/events-toolkit`

Objective: Implement a robust Event Discovery system that automatically generates service manifests, including references to payload schemas, and exposes full schemas via a dedicated endpoint (auto-generated from DTO classes, similar to Swagger).

## Tasks

### 1. Discovery Module Setup [DONE]

- Create `src/discovery/discovery.module.ts`
- Create `src/discovery/discovery.service.ts`
- Add configuration options in `EventsToolkitModule.forRoot()`:

     ```ts
     discovery: {
       enabled: boolean;                    // default: true
       registerOnStartup: boolean;          // default: true
       heartbeatIntervalMinutes: number;    // 0 = disabled, >0 = interval
       includeFullManifestInHeartbeat: boolean; // default: false
     }
     ```

### 2. Service Manifest & Schema References [DONE]

- Create `src/discovery/dto/service-manifest.dto.ts`
- In `consumes[]` and `produces[]` include:

  - `payloadSchemaRef`: string (e.g. `"PaymentProofUploadedEvent"` or `"#/components/schemas/PaymentProofUploadedEvent"`)
  - `description`, `version`, `handler`, `tags`, etc.

- Implement `ManifestService` that:
  - Uses NestJS reflection to scan `@OnEvent`, `@EmitEvent`, and `@OnRequestReply`
  - Extracts metadata + `payloadSchemaRef` from the Event classes (DTOs)

### 3. Auto-generation of JSON Schemas from DTOs

- Create utility `src/discovery/utils/schema-generator.ts`
- Implement logic to auto-generate JSON Schema from classes decorated with `class-validator` (similar to how Swagger generates OpenAPI schemas).
- Support one or more `examples` per schema (can be defined via decorator or static property in the DTO).
- Cache the generated schemas.

### 4. Enhance Existing Decorators

- Update `@OnEvent()`, `@EmitEvent()` and `@OnRequestReply()` to accept rich metadata:

     ```ts
     @OnEvent('payment.proof.uploaded', {
       description: "...",
       tags: ["payment"],
       payloadExample: { ... }   // optional
     })
     ```

- Update documentation with clear instructions for AI agents / developers on how to properly document events.

### 5. Service Information

- Automatically read `name`, `version`, and `description` from `package.json`
- Generate `instanceId`
- Allow overrides via config

### 6. Automatic Registration Events

- On startup (`OnApplicationBootstrap`): emit `platform.service.register.v1` with full manifest (including `payloadSchemaRef`)
- On shutdown: emit lightweight shutdown event
- Heartbeat (if enabled): lightweight "alive" event (`platform.service.heartbeat.v1`). Full manifest only if explicitly configured

### 7. HTTP Endpoints

- Create `src/discovery/discovery.controller.ts`
- Implement two endpoints:
  - `GET /manifest` → Returns the service manifest (with `payloadSchemaRef`, descriptions, etc.)
  - `GET /schemas` → Returns all JSON Schemas used by this service (full schemas with examples)

### 8. Testing Support for Discovery

- Extend the existing `src/testing/` module with specific helpers for Discovery features:
  - Create `MockManifestService`
  - Create `MockDiscoveryService`
  - Add utilities to capture and assert `platform.service.register.v1` and `platform.service.heartbeat.v1` events emitted during tests
  - Provide a method to manually trigger manifest generation in tests (`generateManifest()`)

- Update `EventsToolkitTestModule.forRoot()` to support Discovery configuration:
  - Allow easy disabling of automatic registration (`discovery: { enabled: false }`)
  - Provide mock implementations for `ManifestService` and `DiscoveryService`
  - Support spying/asserting on emitted discovery events

- Add clear testing examples (link them in the README file) showing:

  ```ts
  it('should emit platform.service.register.v1 on startup with correct manifest', async () => {
    const events = mockProducer.getPublishedEventsByType('platform.service.register.v1');
    expect(events.length).toBe(1);
    expect(events[0].data.service).toBe('payment-service');
    expect(events[0].data.consumes.length).toBeGreaterThan(0);
  });
  ```

- Ensure all new Discovery endpoints (`/manifest`, `/schemas`) can be easily tested with the test module.

### 9. Documentation Updates

- Add new doc file with "Event Discovery & Service Registry", and link it to README and related files.
- Include in new doc file:
  - How manifests are built and how `payloadSchemaRef` works
  - Explanation of the two endpoints (`/manifest` and `/schemas`)
  - How schemas are auto-generated from DTOs (class-validator)
  - Full example of `platform.service.register.v1` payload
  - Integration with future `ms-discovery` service:
    - `ms-discovery` will listen to `platform.service.register.v1`
    - It will collect manifests and schemas from all services
    - It will generate the global AsyncAPI specification and event catalog
  - Guidelines for developers and AI agents on how to properly annotate events

- Update `docs/event-convention.md` with the new `platform.*` subjects.

## Acceptance Criteria

- Manifest includes `payloadSchemaRef` in consumes/produces
- `/manifest` and `/schemas` endpoints work correctly
- Schemas are auto-generated from DTO classes
- Discovery can be enabled/disabled and configured via `forRoot()`
- Documentation clearly explains the full flow and future integration with `ms-discovery`
