# Refactor Decorator Options + Comprehensive Documentation Overhaul

**Context:** The decorator option interfaces (`EmitEventOptions`, `OnEventOptions`, `OnRequestReplyOptions`) currently have all fields optional. Downstream, `ManifestEntryBuilder` fills missing `version` and `description` with fallback defaults (`?? '1'`, `?? ''`), while `tags` defaults to `[]`. The documentation guidelines already say to always provide `description` and `tags`, but the types don't enforce this.

After tightening the types, the entire documentation surface must be reviewed and improved so that an AI agent importing this library has a clear, step-by-step process on how to correctly use it in a microservice — from installation through production deployment.

---

## Tasks

### 1. Make `version`, `description`, `payloadExample` Required in Decorator Options [DONE]

**Files to modify:**

- `src/producer/decorators/emit-event.decorator.ts` — `EmitEventOptions`
- `src/consumer/decorators/on-event.decorator.ts` — `OnEventOptions`
- `src/consumer/decorators/on-request-reply.decorator.ts` — `OnRequestReplyOptions`

**Changes per interface:**

| Field | Current | New |
|-------|---------|-----|
| `version` | `version?: string` | `version: string` (emit-event, on-event only — NOT on request-reply) |
| `description` | `description?: string` | `description: string` |
| `payloadExample` | `payloadExample?: Record<string, unknown>` | `payloadExample: Record<string, unknown>` |
| `tags` | `tags?: string[]` | keep optional (empty array is a valid default) |
| `payloadSchemaRef` | `payloadSchemaRef?: string` | keep optional (auto-resolved from reflect metadata) |
| `companyId` | `companyId?: string` | keep optional (request-reply only, tenant filter) |

**Note:** `@OnRequestReply` does NOT have a `version` field today (neither in its interface nor in the metadata). Leave it absent.

**Test:** review if any test needs to be updated, or new ones to be defined.

### 2. Remove Fallbacks for `version` and `description` in `ManifestEntryBuilder` [DONE]

**File:** `src/discovery/manifest-entry.builder.ts`

**Changes:**

```typescript
// Current - `buildOnEventEntry()`:
const version = metadata.version ?? '1';          // remove fallback
description: metadata.description ?? '',            // remove fallback
tags: metadata.tags ?? [],                          // KEEP this fallback

// Current - `buildOnRequestReplyEntry()`:
description: metadata.description ?? '',            // remove fallback
tags: metadata.tags ?? [],                          // KEEP this fallback
// (no version field - version is hardcoded '1')

// Current - `buildEmitEventEntry()`:
const version = metadata.version ?? '1';          // remove fallback
description: metadata.description ?? '',            // remove fallback
tags: metadata.tags ?? [],                          // KEEP this fallback
```

After the change the fields are guaranteed non-null by the type system (for version/description). Keep `tags` fallback since `tags` remains optional.

#### 6.2 Define the Onboarding Flow for an AI Agent

1. **Understand the architecture** - one-page summary of NATS + JetStream, the event envelope, actors, tenant isolation
2. **Install and configure** - `npm install`, `EventsToolkitModule.forRoot()`, NATS connection options
3. **Define an event DTO** - extend `EventBase<T>`, add `class-validator` decorators
4. **Produce an event** - `@EmitEvent()` decorator, `ProducerService.emit()`, the interceptor flow
5. **Consume an event** - `@OnEvent()` decorator, handler signature, DLQ routing
6. **Request-reply** - `sendRequest()` / `request()`, `@OnRequestReply()`, response naming conventions
7. **Outbox pattern** - `@OutboxEvent()`, `OutboxService`, scheduled flushing
8. **Service discovery** - how manifests are generated, `GET /discovery/manifest`, platform events
9. **Schema generation** - auto JSON Schema from DTOs, `payloadSchemaRef` resolution
10. **Testing** - `EventsToolkitTestModule`, mocking NATS, writing unit/integration tests
11. **Deployment** - JetStream stream configuration, environment variables, health checks
