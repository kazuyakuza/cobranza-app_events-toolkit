# Fix `@cobranza-apps/events-toolkit` Module Export Bug

**Library:** `@cobranza-apps/events-toolkit`  
**Repository:** `https://github.com/kazuyakuza/cobranza-app_events-toolkit`  
**Affected Version:** `0.7.3`  
**Discovered In:** `ms-db-gateway` during `npm run test` and `nest start`  

## Problem

`EventsToolkitModule.forRoot()` and `EventsToolkitModule.forRootAsync()` export service tokens (`ProducerService`, `ConsumerService`, `OutboxService`, `DiscoveryService`, `EventLoggerService`) that are **not** present in the module's own `providers` array.

Those services are actually providers of imported sub-modules (`ProducerModule`, `ConsumerModule`, etc.), each declared with `global: true`. NestJS 11's `Module.validateExportedProvider` rejects exporting a token that is neither in the module's own `providers` nor directly re-exported from an `imports` entry.

This causes:

- `Nest cannot export a provider/module that is not a part of the currently processed module (EventsToolkitModule)`
- **Production startup failure** (`nest start` fails)
- **Test compilation failure** (`Test.createTestingModule` fails)

## Root Cause Location

`dist/events-toolkit.module.js` (source: `src/events-toolkit.module.ts`):

```ts
return {
  module: EventsToolkitModule,
  imports: [ProducerModule, ConsumerModule, OutboxModule, DiscoveryModule],
  providers: [eventsToolkitOptionsProvider, jetstreamTokenProvider, eventLoggerServiceProvider],
  exports: [ProducerService, ConsumerService, OutboxService, EventLoggerService, DiscoveryService],
};
```

`ProducerService` etc. are in `ProducerModule.providers`, not in `EventsToolkitModule.providers`.

## Tasks

### 1. Fix `forRoot` and `forRootAsync` exports

**Options (choose one):**

- **Option A — Remove redundant exports:** Since `ProducerModule`, `ConsumerModule`, `OutboxModule`, `DiscoveryModule` are already `global: true`, their providers are available application-wide. The `exports` array in `EventsToolkitModule` is redundant. **Remove the `exports` array entirely** from both `forRoot` and `forRootAsync`.

- **Option B — Re-export via module references:** Change `imports` to also re-export the sub-modules:

  ```ts
  imports: [ProducerModule, ConsumerModule, OutboxModule, DiscoveryModule],
  exports: [ProducerModule, ConsumerModule, OutboxModule, DiscoveryModule],
  ```

  This is the standard NestJS pattern for re-exporting.

- **Option C — Add local providers:** Keep `exports` but add corresponding `useExisting` providers to the module's own `providers`:

  ```ts
  providers: [
    ...existingProviders,
    { provide: ProducerService, useExisting: ProducerService },
    // ... etc
  ],
  exports: [ProducerService, ConsumerService, OutboxService, EventLoggerService, DiscoveryService],
  ```

**Recommendation:** Option A is simplest and correct because the sub-modules are `global: true`. Option B is also valid if consumers expect `EventsToolkitModule` to explicitly re-export.
