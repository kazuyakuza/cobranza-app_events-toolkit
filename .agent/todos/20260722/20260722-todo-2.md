# TODO List #7 – Idempotency Support

Library: `cobranza-apps/events-toolkit`

Objective: Provide consistent, configurable idempotency helpers that follow the same design patterns as the Outbox module (repository pattern, support for PostgreSQL and SQLite, configurable via `forRoot()`).

IMPORTANT: this file only contains proposals that **don't take on account current code** (I mean, source code was not read to write down this file). So, anything here can be used only as a guide/idea, and can (and must) be changed to correctly follow current implementation.

## Tasks

1. Idempotency Module Setup
   - Create `src/idempotency/idempotency.module.ts`
   - Create `src/idempotency/idempotency.service.ts`
   - Add configuration options in `EventsToolkitModule.forRoot()`:

     ```ts
     idempotency: {
       enabled: boolean;                    // default: true
       type: 'postgres' | 'sqlite' | 'redis' | 'memory';  // 'memory' for tests only
       // type-specific options (e.g. ttlSeconds, connection details, sqlite path, etc.)
     }
     ```

2. Repository Pattern
   - Define interface `IdempotencyRepository` with methods:
     - `isProcessed(eventId: string, correlationId?: string): Promise<boolean>`
     - `markAsProcessed(eventId: string, correlationId?: string, ttlSeconds?: number): Promise<void>`
     - `clearExpired()` (optional, for maintenance)
   - Implement concrete repositories:
     - `PostgresIdempotencyRepository`
     - `SqliteIdempotencyRepository`
     - `RedisIdempotencyRepository` (recommended for high-throughput)
     - `MemoryIdempotencyRepository` (for testing)

3. IdempotencyService
   - Main service that uses the configured repository
   - Provide convenient methods:

     ```ts
     await this.idempotencyService.isDuplicate(event);           // uses id + correlation_id
     await this.idempotencyService.markAsProcessed(event);
     await this.idempotencyService.executeIfNotProcessed(event, async () => { ... });
     ```

4. Integration with ConsumerService
   - Add optional automatic idempotency check in `JetStreamConsumerService` when a decorator flag is used:

     ```ts
     @OnEvent('payment.proof.uploaded', { idempotent: true })
     ```

   - Or manual usage inside handlers (recommended for more control).

5. Integration with Discovery (optional)
   - Add `"idempotency"` to the `capabilities` array in the service manifest if enabled.

6. Documentation & Examples
   - Add section in README.md: "Idempotency Support"
   - Explain:
     - Why idempotency is important
     - How to configure (PostgreSQL vs SQLite vs Redis)
     - Recommended usage patterns (manual vs automatic)
     - Best practices for key generation (`event.id + correlation_id`)
   - Provide clear examples for both simple and complex use cases.

7. Testing Support
   - Extend `src/testing/` with `MockIdempotencyService`
   - Update `EventsToolkitTestModule` to easily mock idempotency behavior

---

# Acceptance Criteria

- Idempotency works consistently with PostgreSQL and SQLite (same as Outbox)
- Configuration is done via `EventsToolkitModule.forRoot()`
- `IdempotencyService` provides both low-level and high-level (convenience) methods
- Documentation clearly explains when and how to use it
- Tests can easily mock idempotency checks
