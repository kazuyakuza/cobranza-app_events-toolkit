# Add INBOX-aware response publishing fallback to @cobranza-apps/events-toolkit [DONE]

## Context

When using `@cobranza-apps/events-toolkit` in a request-reply pattern with `nats req` (core NATS), the caller sets a `reply_to` field in the envelope pointing to an INBOX subject (e.g., `INBOX.manual.company.create.550e8400`). The toolkit's `RequestReplyService.sendResponse()` publishes the response via `ProducerService.publish()`, which uses `jetStream.publish()`. INBOX subjects do not match any JetStream stream, so the publish times out waiting for a PubAck. The `RequestReplyMessageProcessor` only acks the inbound message after `dispatch()` succeeds; since `sendResponse()` throws, the message is never acked, and JetStream redelivers it repeatedly. This causes duplicate processing on the consumer side.

This was worked around in `ms-db-gateway` by detecting INBOX subjects in `executeNatsDispatch` and using core NATS `publish` directly. The fix should live in the library so all consumers benefit.

## Library Project

`https://github.com/kazuyakuza/cobranza-app_events-toolkit`
Current version: `0.12.0`

## Root Cause in the Toolkit

1. `ProducerService.publish(subject, event)` always uses `this.jetStream.publish(subject, encodedPayload)`
2. `RequestReplyService.sendResponse(correlationId, responseEvent)` calls `this.producerService.publish(replyTo, responseEvent)`
3. `RequestReplyMessageProcessor.processMessage(msg, subject)` calls `msg.ack()` only after `await this.dispatch(dispatchOptions)` resolves successfully
4. If `dispatch()` internally calls `sendResponse()` and it throws, the exception bubbles to `processMessage()`, which does NOT ack — it may nack or rethrow, causing JetStream redelivery

## Design Options

### Option A: `fallbackToCoreNatsOnInbox` in `RequestReplyConfig` (Recommended)

Add a boolean config flag `fallbackToCoreNatsOnInbox` (default `false` for backward compatibility, or `true` for next minor).

When enabled, `RequestReplyService.sendResponse()` detects if `replyTo` matches the INBOX pattern (`^(_)?INBOX\.`). If so, it uses `this.natsConnection.publish(replyTo, encodeEvent(responseEvent))` directly instead of going through `ProducerService.publish()`.

**Pros:**

- Targeted fix, minimal surface area
- Backward compatible
- No behavior change for non-INBOX responses
- Easy to understand and configure

**Cons:**

- Only fixes INBOX subjects, not other non-stream subjects
- Requires opt-in (or a breaking default change)

### Option B: `responseTransport: 'jetstream' | 'core-nats' | 'auto'` in `RequestReplyConfig`

Add a transport mode enum. `'auto'` tries JetStream first, falls back to core NATS on failure.

**Pros:**

- More general than Option A
- 'auto' handles any unmatched-stream subject

**Cons:**

- Fallback-by-exception is fragile (different NATS versions throw different errors)
- Slightly more complex

### Option C: Ack before `sendResponse` in `RequestReplyMessageProcessor`

Change `processMessage()` to `msg.ack()` immediately after `dispatch()` resolves, then call `sendResponse()` in a fire-and-forget manner.

**Pros:**

- Prevents redelivery regardless of response transport
- Simplest code change

**Cons:**

- Changes reliability semantics: if response publish fails, the message is gone but the caller never gets a response
- Not appropriate for all use cases

### Option D: Add core-NATS fallback inside `ProducerService.publish()`

When `jetStream.publish()` fails, catch and retry with `this.natsConnection.publish()`.

**Pros:**

- Fixes the issue for ALL publishes, not just responses
- Centralized in one place

**Cons:**

- Broad impact — changes behavior for all producers
- Determining "should fall back" from arbitrary errors is fragile
- Could mask real stream configuration issues

## Recommended Implementation: Option A

Use Option A as the primary fix. Optionally combine with a targeted portion of Option C: ensure `processMessage` handles `sendResponse` errors gracefully (nack vs ack depending on error type), but that is a larger semantic change and should be a separate task.

### 2. `src/request-reply/request-reply.service.ts`

Modify `sendResponse()`:

```typescript
async sendResponse(correlationId: string, responseEvent: AnyEventEnvelope<unknown>): Promise<void> {
  const replyTo = responseEvent.reply_to;
  ensureReplyTo(replyTo, correlationId);

  if (this.shouldUseCoreNats(replyTo)) {
    this.natsConnection.publish(replyTo, encodeEvent(responseEvent));
    return;
  }

  await this.producerService.publish(replyTo, responseEvent);
}

private shouldUseCoreNats(replyTo: string): boolean {
  if (!this.config.fallbackToCoreNatsOnInbox) return false;
  return isInboxSubject(replyTo);
}
```

Add `isInboxSubject` helper (could be in a shared utils file):

```typescript
function isInboxSubject(subject: string): boolean {
  return /^_?INBOX\./.test(subject);
}
```

Import `encodeEvent` from serialization utils (it may already be imported transitively via `ProducerService`).
