# Plan: Fix Critical Workflow Bypass & Tool Selection Non-Compliance

**Supersedes** prior analysis. Root cause from `.kilo/plans/1779807580595-cosmic-comet.md`.

## Two Distinct Problems

### Problem 1: Critical Workflow Bypass (Primary)

The Plan Agent:
1. Generates global plan → gets user approval
2. **Calls `plan_exit`** → transitions from Plan mode to Code mode
3. Code-mode agent receives TODO list + global plan but has **no active workflow context**
4. Code agent implements everything directly — **skips all 4.x sub-steps** (analysis, review, docs, verification, individual [DONE] marking)

**Root cause**: `plan_exit` is called after plan approval, ending Plan Agent orchestration prematurely. Critical Workflow requires Plan Agent to continue orchestrating all 4.x sub-steps via `task` tool. No instruction explicitly forbids `plan_exit` before Step 6 completes.

### Problem 2: Tool Selection Priority Ignored (Secondary)

Sub-agents use `bash` / `powershell` for file manipulation and code operations instead of:
- `vscode-mcp-server_*` tools (rename, replace_lines, create_file, move_file)
- `Bifrost_*` tools (semantic find, rename, definitions)

**Root cause**: Sub-agent system prompts do not reference `tool-selection-priority.md`. The rule exists but is only visible to the Plan Agent via `kilo.jsonc` instructions glob. Sub-agents never see it.

## Solution

### Fix A — Prevent `plan_exit` During Critical Workflow

**File**: `.kilo/commands/critical-workflow.md`

Add to Section 4.0 (after existing CRITICAL line about task delegation):

```markdown
- **CRITICAL**: Do NOT call `plan_exit` at any point during this workflow. The Plan Agent
  remains in Plan mode as orchestrator for the entire TODO file lifecycle (Steps 1–6).
  All delegation to sub-agents happens exclusively via the `task` tool. `plan_exit` is
  only safe to call after Step 6 is fully complete.
```

**Rationale**: This is Solution C from cosmic-comet.md. Prevents the exact failure mode that occurred.

### Fix B — Add Compliance Self-Check

**File**: `.kilo/commands/critical-workflow.md`

Add to Section 4.0:

```markdown
- **Compliance Self-Check**: Before executing any 4.x sub-step, the agent MUST verify:
  (a) Am I still the Plan Agent orchestrating via `task` tool?
  (b) Is this a single discrete sub-step assigned to the correct sub-agent type?
  If either answer is "no", stop and re-read this workflow from the beginning.
```

**Rationale**: Runtime guard that catches scope creep before it happens.

### Fix C — Add Re-Entry Procedure to Step 6

**File**: `.kilo/commands/critical-workflow.md`

Add to Section 6 (Continuation):

```markdown
- **New Session Re-entry**: When a new session starts for Step 6 continuation:
  1. Re-read this workflow document in full.
  2. Read the saved global plan from `.kilo/plans/`.
  3. Read `.kilo/state.json` to find the last completed sub-step.
  4. Resume from the next incomplete sub-step via `task` tool delegations.
  5. Do NOT execute tasks directly.
```

**Rationale**: Solution B from cosmic-comet.md. Handles legitimate new-session scenarios.

### Fix D — Mandatory Sub-Task Prompt Template

**File**: `.kilo/commands/critical-workflow.md`

Add new section after 4.0 titled "Sub-Task Prompt Requirements":

```markdown
#### Sub-Task Prompt Requirements

Every `task` tool invocation from the Plan Agent MUST prepend the following to the prompt:

---
SUB-AGENT TASK — SINGLE DISCRETE STEP
- You are executing exactly ONE step of a larger Critical Workflow plan.
- Do ONLY what is described below. Do NOT execute subsequent steps.
- Do NOT read or expand scope to the global plan for other tasks.
- Follow the Tool Selection Priority rule: prefer vscode-mcp-server_* and Bifrost_*
  tools over bash for code operations. Reserve bash for git, npm, builds, tests.
- Signal completion with a clear summary: what was done, what was NOT done.
- If anything is ambiguous or outside your assigned scope, return the question
  to the caller. Do NOT make assumptions.
---
```

**Rationale**: Ensures every sub-agent receives explicit scope boundaries and tool preferences directly in its prompt, regardless of its agent definition.

### Fix E — Update Agent Definitions with Tool Preferences

**Files**: `.kilo/agents/code-reviewer.md`, `.kilo/agents/code-simplifier.md`, `.kilo/agents/docs-specialist.md`, `.kilo/agents/frontend-specialist.md`

For each file, add under the identity line:
```markdown
Tool Preference: Always prefer vscode-mcp-server_* and Bifrost_* tools for code
operations. Use bash ONLY for git, npm, builds, and tests. See
.kilo/rules/tool-selection-priority.md.
```

Preserve all existing frontmatter, permissions, and role-specific instructions.

**Rationale**: Tool preference must be in the sub-agent's system prompt to be effective. The prompt template (Fix D) covers the dynamic prompt, but agent definitions provide the static baseline. This is NOT about "scope boundaries" — those are handled by Fix D in the prompt.
