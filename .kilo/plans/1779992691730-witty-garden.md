# Fix Critical Workflow — TODO Detection, State Ownership, History Cleanup

## Problem Statement

The `critical-workflow.md` has three operational bugs discovered during live testing with `.agent/todos/20260521-todo-1.md`:

1. **TODO task under-detection** — Section-based TODOs with `### N. Title` task headers are not being parsed into distinct tasks, causing the global plan to contain a single 4.x cycle instead of one per task.
2. **State.json edit permission noise** — Sub-agents are instructed to update `.kilo/state.json`, triggering permission prompts. Only the Plan Agent should touch this file.
3. **History accumulation** — The `history` array in `state.json` is never cleared between global plans, causing stale data and confusing resumption logic.

## Proposed Fixes

### Fix 1: Explicit Section-Based TODO Parsing Rules

Update the **TODO File Format** block (Step 1) to replace vague "interpret the file" instructions with exactly three deterministic patterns. Each pattern produces a list of tasks; sub-items (bullets, checklists, nested paragraphs) under a task heading belong to that task and do **not** spawn new tasks.

- **Pattern A — Root Sections as Tasks**: If the file has no single top-level `# Title` and instead begins directly with multiple `# Heading` sections, each `# Heading` is a task.
- **Pattern B — Sub-Sections as Tasks**: If the file has one top-level `# Title` followed by multiple `## Heading` sections, each `## Heading` is a task.
- **Pattern C — Tasks Section Children as Tasks**: If the file has a top-level `# Title`, then `##` sub-sections, and one of those `##` sub-sections is named **Tasks** (case-insensitive, also accept *Tareas*, *Task List*, etc.), then each `### Heading` inside that Tasks section is a task.

**Disambiguation rule**: Apply the patterns in order (A → B → C). Use the first pattern that matches. If none match, ask the user BEFORE generating the global plan.

**Example mapping** for `20260521-todo-1.md` (matches Pattern C):
- `### 1. Create Problem Section Component` → Task 1 (sub-items are implementation details for this task)
- `### 2. Create Solution Section Component` → Task 2
- `### 3. Integrate into Main App` → Task 3
- `### 4. Styling & Polish` → Task 4

### Fix 2: State Ownership (Plan Agent Only)

Add a new subsection under **State Tracking**:

```
### State Ownership

- ONLY the Plan Agent reads or writes `.kilo/state.json`.
- Sub-agents MUST NOT edit, create, or reference `state.json`.
- When a sub-agent completes its step, it reports completion in its return message.
- The Plan Agent then updates `state.json` before dispatching the next sub-task.
```

**Rationale for permission issues**: Because sub-agents never touch `state.json`, no per-model permission configuration is required for this file. All `state.json` I/O is concentrated in the Plan Agent, eliminating cross-mode permission prompts entirely.

Update **Step 4.6** to remove the instruction for the implementer sub-agent to update `state.json`; instead, the implementer marks the task as `[DONE]` in the TODO file and commits, then signals completion. The Plan Agent updates `state.json` after receiving the signal.

### Fix 3: State Lifecycle & History Cleanup

Extend **State Tracking** with lifecycle rules:

- **On new global plan start** (Step 1, before generating the plan):
  - If `current_todo_file` is `null` or different from the TODO file now being processed, reset `history` to `[]` and set `current_todo_file` to the new file path.
  - Reset `current_task` to `{ index: 0, description: "...", sub_step: "2.0", sub_step_status: "PENDING", attempts: 0 }`.

- **On global plan finish** (Step 5, after TODO file renamed and committed):
  - Clear `history` to `[]`.
  - Set `current_todo_file` to `null`.
  - Set `current_task.sub_step_status` to `"COMPLETED"`.

- **On resumption with stale/empty state** (Step 6, New Session Re-entry):
  - After reading `state.json`, if `current_task.sub_step` is `"1.0"` or `current_todo_file` is `null`, read the target TODO file.
  - Count how many sections already have `[DONE]` in their title.
  - Set `current_task.index` to the first undone task and `sub_step` to `"4.1"` (ready for planning) or `"4.2"` if a plan file already exists for that task.

### Fix 4: Update Step References

- In **Step 4.0**, replace "State Sync: when committed, update `.kilo/state.json`..." with "State Sync: the Plan Agent updates `.kilo/state.json` after each sub-agent signals completion."
- In **Step 4.6**, remove "update `.kilo/state.json` setting..." from the implementer's duties; add a Plan Agent note immediately after the implementer step: "Plan Agent updates `state.json` to `sub_step: '4.6'` and `sub_step_status: 'COMPLETED'` before proceeding."
- In **Step 5**, add: "Plan Agent clears `state.json` history and resets `current_todo_file` to `null`."
