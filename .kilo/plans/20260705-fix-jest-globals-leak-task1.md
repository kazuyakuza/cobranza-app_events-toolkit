## 1. Problem Statement

The package main entry `dist/index.js` transitively loads `dist/testing/assertion.helpers.js` and `dist/testing/discovery-assertion.helpers.js`, both of which `require('@jest/globals')`. Any consumer importing from the main entry outside a Jest environment crashes with:

```
Do not import `@jest/globals` outside of the Jest test environment
```

## 2. Module Graph Audit (Verified)

### 2.1 `@jest/globals` import locations

Grep across `src/` confirms exactly **two** source files import `@jest/globals`, both inside `src/testing/`:

| File | Line | Statement |
|------|------|-----------|
| `src/testing/assertion.helpers.ts` | 1 | `import { expect } from '@jest/globals';` |
| `src/testing/discovery-assertion.helpers.ts` | 1 | `import { expect } from '@jest/globals';` |

No file outside `src/testing/` imports `@jest/globals`.

### 2.2 Path from main entry to `./testing`

Grep confirms the **only** reference to `./testing` from anywhere in `src/` is in `src/index.ts`:

- `src/index.ts` line 40: `export * from './testing';`

No other source file imports from `./testing` or `../testing`. Therefore **the sole path from the main module graph to `@jest/globals` is `src/index.ts` → `./testing` → `assertion.helpers.ts` / `discovery-assertion.helpers.ts`**.

### 2.3 Build output

- `tsconfig.build.json` excludes `**/*.spec.ts`, `**/*.test.ts`. Assertion helpers are `.ts` sources, so they compile into `dist/testing/`.
- Build command: `npm run build` → `tsc -p tsconfig.build.json` → emits `dist/` with `dist/index.js`, `dist/testing/index.js`, etc.
- Currently `dist/` is not present (cleaned). It will be regenerated by the build step.

### 2.4 `package.json` current state

- `"main": "dist/index.js"` ✅
- `"types": "dist/index.d.ts"` ✅
- **No `"exports"` field** — needs to be created.
- `@jest/globals` is correctly listed under `devDependencies` (not runtime) — confirms it should never reach a non-Jest consumer.

### 2.5 Conclusion

Removing the `export * from './testing';` line from `src/index.ts` fully severs the leak path. Testing symbols must then be reachable only through a dedicated subpath export `@cobranza-apps/events-toolkit/testing`.

---

## 3. High-Level Approach

1. **Edit `src/index.ts`** — remove `export * from './testing';` (the only source of the leak).
2. **Edit `package.json`** — add an `exports` map exposing `.` (main entry) and `./testing` (testing subpath), each with `types` + `default` conditions, so TypeScript and Node both resolve the subpath. Keep `main`/`types` for legacy non-`exports` resolvers.
3. **Build** and verify the leak is gone.
4. **Add a regression spec** that asserts `require('./index')` (after build) does not transitively load `@jest/globals`, and that the testing subpath still loads correctly in a Jest context.
5. **Update docs** (`README.md` "Testing Utilities" section + `docs/testing-utilities.md`) to instruct consumers to import testing utilities from `@cobranza-apps/events-toolkit/testing`.
6. **Run lint/typecheck/tests** and commit.

No internal `src/` file needs to change its import paths — none currently import from `./testing` or `../testing` except `src/index.ts` itself.

### Step 4.6 — Add a regression test

**New file:** `src/entry-point-isolation.spec.ts`

This spec runs inside Jest (`testEnvironment: 'node'` per `jest.config.js`), so `@jest/globals` is available and the testing subpath can be required. It asserts that requiring the **compiled main entry** does not pull in `@jest/globals` and does not pull in the testing module.

```ts
import { existsSync } from 'fs';
import { join } from 'path';
import { expect, describe, it } from '@jest/globals';

describe('main entry isolation from @jest/globals', () => {
  it('dist/index.js exists after build', () => {
    const distIndex = join(process.cwd(), 'dist', 'index.js');
    expect(existsSync(distIndex)).toBe(true);
  });

  it('dist/index.d.ts exists after build', () => {
    const distTypes = join(process.cwd(), 'dist', 'index.d.ts');
    expect(existsSync(distTypes)).toBe(true);
  });

  it('dist/testing/index.js exists after build', () => {
    const distTesting = join(process.cwd(), 'dist', 'testing', 'index.js');
    expect(existsSync(distTesting)).toBe(true);
  });

  it('main entry does not transitively load @jest/globals', () => {
    const distIndex = join(process.cwd(), 'dist', 'index.js');

    delete require.cache[require.resolve(distIndex)];

    const mainModule = require(distIndex);
    expect(mainModule).toBeDefined();
    expect(typeof mainModule.EventsToolkitModule).toBe('function');

    const testingKeySep = join('dist', 'testing') + '/';
    const testingKeyWin = join('dist', 'testing') + '\\';
    const testingCacheKey = Object.keys(require.cache).find((key) =>
      key.includes(testingKeySep) || key.includes(testingKeyWin),
    );
    expect(testingCacheKey).toBeUndefined();
  });

  it('testing subpath is loadable inside a Jest environment', () => {
    const distTesting = join(process.cwd(), 'dist', 'testing', 'index.js');
    const testingModule = require(distTesting);
    expect(testingModule).toBeDefined();
    expect(typeof testingModule.MockProducerService).toBe('function');
    expect(typeof testingModule.expectEventPublished).toBe('function');
  });
});
```

**Placement:** at `src/` root (NOT inside `src/testing/`) so it is not itself part of the testing barrel and does not pull `@jest/globals` into the main graph. `jest.config.js` `testRegex: .*\.spec\.ts$` and `rootDir: 'src'` will pick it up.

**Dependency note:** This spec must run **after** `npm run build` so `dist/` exists. The cleanest approach is to add a `pretest` hook (see step 4.7).

**Windows-path note:** The cache-key check uses BOTH backslash and forward-slash variants so it works on Windows (`dist\testing\...`) and POSIX CI.
