## Issue 3 — README async request-reply example uses inconsistent response subject

**File:** `README.md`  
**Lines:** ~458, ~515

The async request-reply example in README sets `replyTo` to a subject built with `action: 'requested.response'`, but the response handler is declared with `@OnRequestReply('credit.check.completed')` and the response envelope `type` is `credit.check.completed`.

Because `RequestReplyConsumerService` subscribes to the wildcard `company.*.response.v1`, the message is delivered, but the example mixes the alternative `.response` suffix convention with the preferred past-tense outcome convention. This is inconsistent with:

- The Task B plan’s stated preference: *“The `DebtService.requestCreditCheck` reply-subject uses `action: 'completed'` for the reply subject — keep (preferred past-tense convention).”*
- `docs/examples/async-request-reply.example.ts`, which already uses `action: 'completed'`.

### Fix

In `README.md`, change the requester’s reply subject from:

```typescript
const replySubject = this.subjectBuilder.build({
  companyId, domain: 'credit', entity: 'check',
  action: 'requested.response', version: '1',
});
```

To:

```typescript
const replySubject = this.subjectBuilder.build({
  companyId, domain: 'credit', entity: 'check',
  action: 'completed', version: '1',
});
```

Do the same for the `sendRequestThroughOutbox` low-level example at line ~621, which currently also uses `action: 'requested.response'`.
